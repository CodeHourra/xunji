## 一、设计决策摘要

| 决策项 | 结论 |
|--------|------|
| 整体架构 | App + 按需 MCP（方案 C） |
| 技术栈 | Tauri 2.0 + Vue 3 + Rust + TypeScript sidecar |
| TS 运行时 | Bun（编译为单独二进制，适配 Tauri sidecar 分发） |
| Python sidecar | 不采用，改用 TS/Bun 替代 |
| 项目结构 | Monorepo：apps/desktop + packages/sidecar + packages/mcp-server + packages/shared |
| Rust ↔ TS 通信 | stdin/stdout JSON-RPC 2.0 |
| MCP Server | stdio transport，AI IDE 按需 spawn，独立二进制 |
| 前端布局 | TopBar（Tab 切换）+ Sidebar（纯筛选）+ Main Content（两栏） |
| 详情页 | 替换列表进入详情页（非三栏 DetailPanel） |

### 三种运行态

| 运行态 | 活跃组件 | 说明 |
|--------|---------|------|
| **App 打开** | Tauri + Rust + Vue + TS Sidecar | 全功能：采集、提炼、搜索、展示 |
| **App 关闭 + AI IDE 使用** | MCP Server (stdio) | AI IDE 按需 spawn，只读查询知识库 |
| **App 关闭 + Launch Agent** | Launch Agent | 轻量后台：仅扫描文件变化写入 SQLite |

### 通信协议

| 通信路径 | 协议 | 数据格式 |
|---------|------|---------|
| Vue ↔ Rust | Tauri invoke (IPC) | 强类型 TS ↔ Rust serde |
| Rust → TS Sidecar | stdin/stdout | JSON-RPC 2.0 |
| AI IDE → MCP Server | stdio | MCP 协议 (JSON-RPC) |
| Launch Agent → SQLite | 直接写入 | rusqlite / better-sqlite3 |

## 四、数据流水线

### 完整数据流

```
数据源文件变化
      │
      ▼
┌─── Rust Collector ───────────────────────────────────┐
│                                                       │
│  1. Watcher / 定时扫描 / 手动触发                       │
│  2. Source Parser（每数据源独立解析器）                   │
│     ┌──────────┬──────────┬──────────────┐            │
│     │ClaudeCode│ Cursor   │ CodeBuddy    │            │
│     │(JSONL)   │(SQLite + │ (JSONL/JSON) │            │
│     │          │ Lexical) │              │            │
│     └────┬─────┴────┬─────┴──────┬───────┘            │
│          ▼          ▼            ▼                     │
│  3. Normalizer → 统一 NormalizedSession 格式             │
│  4. Deduplicator（去重）                                │
│     session_id 匹配 → 新/更新/相同                      │
└───────────────────────────────────────────────────────┘
      │  event: "sessions-updated"
      ▼
┌─── Rust → TS Sidecar (JSON-RPC) ─────────────────────┐
│                                                       │
│  5. Pre-processor（清理 thinking/tool_call，截断）       │
│  6. Value Judge（Prompt B 轻量版）→ none/low 跳过       │
│  7. Full Distill（Prompt B 完整 + Prompt A 可选）       │
│  8. 结果返回 Rust → 写入 SQLite                        │
└───────────────────────────────────────────────────────┘
```

### Rust ↔ TS Sidecar JSON-RPC 协议

```
Rust（客户端）→ TS Sidecar（服务端）

"ping"         → { status, provider, model }
"judge_value"  → { value: "high"|"medium"|"low"|"none", reason }
"distill_full" → { title, type, value, summary, note, tags, tech_stack, memory?, skill? }
```

### 批量分析并发控制

- Rust 侧使用 Semaphore 控制并发（默认 3，可配置 1-10）
- 每条结果实时落库，不等全部完成
- 单条失败跳过继续，失败可单独重试（最多 2 次，指数退避）

### 去重策略

| 场景 | 检查字段 | 动作 |
|------|---------|------|
| 首次导入 | `session_id` 不存在 | INSERT |
| 增量同步无变化 | `session_id` 存在 + `message_count` 不变 | SKIP |
| 会话有新消息 | `session_id` 存在 + `message_count` 变大 | UPDATE `has_updates = true` |
| 本地+远端同源 | `session_id + source_host` 组合键 | SKIP if exists |
| 兜底 | `content_hash` (MD5) | SKIP if exists |

## 五、数据库 Schema

```sql
CREATE TABLE sources (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    enabled     INTEGER DEFAULT 1,
    scan_paths  TEXT,                  -- JSON array
    last_sync   TEXT,
    config      TEXT
);

CREATE TABLE sessions (
    id              TEXT PRIMARY KEY,
    source_id       TEXT NOT NULL,
    session_id      TEXT NOT NULL,
    source_host     TEXT DEFAULT 'local',
    project_path    TEXT,
    project_name    TEXT,
    message_count   INTEGER DEFAULT 0,
    content_hash    TEXT,
    raw_path        TEXT,
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL,
    status          TEXT DEFAULT 'pending',
    value           TEXT,
    has_updates     INTEGER DEFAULT 0,
    analyzed_at     TEXT,
    error_message   TEXT,
    UNIQUE(session_id, source_host)
);

CREATE TABLE messages (
    id          TEXT PRIMARY KEY,
    session_id  TEXT NOT NULL,
    role        TEXT NOT NULL,
    content     TEXT NOT NULL,
    timestamp   TEXT,
    tokens_in   INTEGER DEFAULT 0,
    tokens_out  INTEGER DEFAULT 0,
    seq_order   INTEGER NOT NULL,
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

CREATE TABLE cards (
    id                TEXT PRIMARY KEY,
    session_id        TEXT NOT NULL,
    title             TEXT NOT NULL,
    type              TEXT,
    value             TEXT,
    summary           TEXT,
    note              TEXT NOT NULL,
    category_id       TEXT,
    memory            TEXT,
    skill             TEXT,
    source_name       TEXT,
    project_name      TEXT,
    prompt_tokens     INTEGER DEFAULT 0,
    completion_tokens INTEGER DEFAULT 0,
    cost_yuan         REAL DEFAULT 0,
    feedback          TEXT,
    created_at        TEXT NOT NULL,
    updated_at        TEXT NOT NULL,
    FOREIGN KEY (session_id) REFERENCES sessions(id)
);

CREATE TABLE tags (
    id      TEXT PRIMARY KEY,
    name    TEXT NOT NULL UNIQUE,
    type    TEXT DEFAULT 'auto',
    count   INTEGER DEFAULT 0
);

CREATE TABLE card_tags (
    card_id TEXT NOT NULL,
    tag_id  TEXT NOT NULL,
    PRIMARY KEY (card_id, tag_id),
    FOREIGN KEY (card_id) REFERENCES cards(id),
    FOREIGN KEY (tag_id)  REFERENCES tags(id)
);

CREATE TABLE categories (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    parent_id   TEXT,
    sort_order  INTEGER DEFAULT 0,
    FOREIGN KEY (parent_id) REFERENCES categories(id)
);

CREATE TABLE sync_log (
    id              TEXT PRIMARY KEY,
    source_id       TEXT NOT NULL,
    started_at      TEXT NOT NULL,
    finished_at     TEXT,
    sessions_found  INTEGER DEFAULT 0,
    sessions_new    INTEGER DEFAULT 0,
    sessions_updated INTEGER DEFAULT 0,
    status          TEXT DEFAULT 'running'
);

CREATE TABLE token_usage (
    id                TEXT PRIMARY KEY,
    card_id           TEXT,
    provider          TEXT NOT NULL,
    model             TEXT NOT NULL,
    prompt_tokens     INTEGER NOT NULL,
    completion_tokens INTEGER NOT NULL,
    cost_yuan         REAL NOT NULL,
    created_at        TEXT NOT NULL,
    FOREIGN KEY (card_id) REFERENCES cards(id)
);

CREATE VIRTUAL TABLE cards_fts USING fts5(
    title, summary, note, tags,
    content='cards',
    content_rowid='rowid',
    tokenize='unicode61'
);

-- 索引
CREATE INDEX idx_sessions_source   ON sessions(source_id);
CREATE INDEX idx_sessions_project  ON sessions(project_name);
CREATE INDEX idx_sessions_status   ON sessions(status);
CREATE INDEX idx_sessions_created  ON sessions(created_at);
CREATE INDEX idx_cards_session     ON cards(session_id);
CREATE INDEX idx_cards_type        ON cards(type);
CREATE INDEX idx_cards_value       ON cards(value);
CREATE INDEX idx_cards_created     ON cards(created_at);
CREATE INDEX idx_card_tags_tag     ON card_tags(tag_id);
CREATE INDEX idx_categories_parent ON categories(parent_id);
```

## 六、TS Sidecar & MCP Server

### Sidecar 架构

```
stdin (JSON-RPC from Rust)
    │
    ▼
  RPC Router
    ├── "ping"         → HealthHandler
    ├── "judge_value"  → ValueJudgeHandler
    ├── "distill_full" → FullDistillHandler
    └── "get_config"   → ConfigHandler
              │
     ┌────────┴────────┐
     ▼                 ▼
 API Provider      CLI Provider
 (HTTP LLM API)   (spawn 本地 CLI)
```

### Distiller 统一接口

```typescript
interface DistillerProvider {
  name: string;
  distill(prompt: string, content: string): Promise<string>;
  isAvailable(): Promise<boolean>;
}
```

- `ApiProvider`：使用 HTTP 调用 LLM API，内置 provider 预设 + openai-compatible 万能模式
- `CliProvider`：spawn 本地 CLI 进程，stdin/stdout 交互

### MCP Server

- 独立二进制（`xunji-mcp`），安装到 `~/.xunji/bin/`
- stdio transport，AI IDE 按需 spawn
- 4 个工具：`xunji_search`、`xunji_get_card`、`xunji_list_tags`、`xunji_add_note`
- 直接读取 SQLite（WAL 模式只读）

用户在 AI IDE 配置：
```json
{
  "mcpServers": {
    "xunji": {
      "command": "~/.xunji/bin/xunji-mcp",
      "env": { "XUNJI_DB": "~/.xunji/db/xunji.db" }
    }
  }
}
```

### Sidebar 四区块

1. **全部对话按钮**：置顶，清除所有筛选

### 会话卡片（三区横条）

左区（25%）元信息 + 中区（55%）Badge/摘要/标签 + 右区（20%）操作按钮

价值等级视觉规则：

| 价值 | 左边框 | Badge | 卡片 | 操作按钮 |
|------|--------|-------|------|---------|
| high | #22c55e 绿 | 绿底白字 `★ high` | 正常+微投影 | 显示全部 |
| medium | #f59e0b 橙 | 橙底白字 `★ mid` | 正常 | 显示全部 |
| low | #d1d5db 灰 | 灰底灰字 `★ low` | opacity: 0.45 | 不显示 |

### 笔记详情页

点击「查看笔记」→ 替换列表进入详情页（带返回按钮）。

内容区：标题 + Badge + Markdown 笔记渲染 + 标签 + 来源信息 + 操作栏

底部信息栏：
- `💬 来自 {source_name} / {project_name} · 📅 {created_at}`
- `提炼消耗：{tokens} tokens · ¥{cost} · 模型：{model}`

底部操作栏（左右分组）：
- 左组：`💬查看对话` `↓导出MD` `🧠生成Memory` `⚡生成Skills` `🔄重新分析`
- 右组：`👍` `👎` `🗑删除笔记` `关闭`

## 八、数据源配置（可配置扫描目录）

Claude Code 等数据源支持配置多个扫描目录，适配官方版、内网版、魔改版：

```toml
# ~/.xunji/config.toml

[[collector.sources]]
id = "claude-code"
name = "Claude Code"
enabled = true
scan_dirs = [
  "~/.claude",
  "~/.claude-internal",    # 内网版 / 魔改版
]

[[collector.sources]]
id = "cursor"
name = "Cursor"
enabled = true
scan_dirs = [
  "~/Library/Application Support/Cursor",
]

[[collector.sources]]
id = "codebuddy-cli"
name = "CodeBuddy CLI"
enabled = false
scan_dirs = [
  "~/.codebuddy",
]
```

v0.1 通过直接编辑 `config.toml` 配置，v0.2 在设置面板提供可视化配置。
